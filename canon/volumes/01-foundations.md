# Volume I: Foundations

## Purpose

This volume defines the basic objects of Cryptographic Reciprocity Theory before they become software mechanisms, economic instruments, or social rituals.

It asks what trust, identity, memory, promise, obligation, reputation, forgiveness, legitimacy, and time mean when actors must cooperate under uncertainty across long horizons.

## Core question

What must be true about actors, memory, promises, and constraints before durable cooperation can be made visible, accountable, and testable?

## Scope

This volume covers foundational concepts. It does not yet design protocols, governance systems, markets, or agent capabilities.

It should give later volumes a shared language so that engineering work does not silently redefine the theory.

## What this volume claims

- Trust is not a static property; it emerges through repeated interaction under uncertainty.
- Identity requires continuity of agency across time.
- Memory is required for obligation, reputation, accountability, and forgiveness.
- Promises become stronger when actors can rely on them and weaker when exit has no cost.
- Reputation compresses memory, but compression can lose context.
- Forgiveness is meaningful only when failure is remembered.
- Legitimacy is distinct from power, signature validity, and technical capability.

## What this volume does not claim

- It does not claim that authentication equals trust.
- It does not claim that a public key is full identity.
- It does not claim that payment is automatically commitment.
- It does not claim that memory alone creates justice.
- It does not claim that reputation can safely be reduced to a single score.
- It does not claim that cryptographic proof preserves human meaning by default.

## Runtime relevance

This volume anchors the vocabulary used by HODLXXI runtime surfaces such as:

- signed capabilities;
- public identity surfaces;
- requester proof paths;
- paid jobs;
- signed receipts;
- attestations;
- local event memory;
- reputation telemetry;
- runtime non-claims.

The runtime should be treated as an evidence generator, not as proof that the foundational claims are true.

## Failure modes

- The runtime proves key control while users interpret it as full identity.
- A receipt proves an event while users interpret it as obligation fulfillment.
- Reputation telemetry becomes mistaken for social reputation.
- Permanent memory creates punishment without repair.
- Forgiveness becomes reputation laundering.
- Legitimacy becomes self-declared capability.

## Falsification focus

This volume should be considered weak wherever its definitions fail to predict observable behavior in HODLXXI runtime, covenant practice, or long-running human-agent interaction.

It should be revised wherever its terms allow users, agents, or institutions to overread cryptographic events as human meaning.
