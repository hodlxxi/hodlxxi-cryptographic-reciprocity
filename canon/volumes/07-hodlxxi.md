# Volume VII: HODLXXI

## Purpose

This volume connects Cryptographic Reciprocity Theory to the actual HODLXXI system.

Theory comes first. Engineering follows as experiment.

## Core question

Can HODLXXI become experimental infrastructure for testing CRT without confusing runtime
evidence with human meaning, obligation fulfillment, or legitimacy?

## Scope

This volume studies the actual HODLXXI runtime, Human Proof, paid jobs, receipts,
attestations, covenant surfaces, agent runtime, marketplace listing, proof-of-funds
surfaces, trust reports, and future delegation or repair mechanisms.

It is the bridge between theory, runtime, observation, and revision.

## What this volume claims

- HODLXXI is behavioral, identity, and trust infrastructure, not merely software.
- HODLXXI mechanisms should test Canon claims rather than merely express them.
- Runtime evidence should be mapped to observable behavior, non-claims, failure modes, and falsification tests.
- Receipts, attestations, reputation telemetry, and identity surfaces are experimental evidence surfaces.
- The system should preserve the distinction between event proof and obligation proof.
- Engineering work should be constrained by Canon language before it becomes runtime authority.

## What this volume does not claim

- It does not claim HODLXXI proves CRT true.
- It does not claim signed receipts prove obligation fulfillment.
- It does not claim public keys are full identities.
- It does not claim reputation telemetry is social reputation.
- It does not claim declared covenants are funded commitments.
- It does not claim current runtime memory includes forgiveness, repair, dispute, or appeal.
- It does not claim agent capabilities are legitimate delegated authority.

## Runtime linkage rule

Every runtime mechanism should eventually be mapped to:

1. a Canon claim;
2. a Lexicon term;
3. an observable behavior;
4. a non-claim;
5. a failure mode;
6. a falsification test;
7. a future correction or experiment.

## Failure modes

- The system proves an action occurred but fails to preserve the meaning of the commitment behind it.
- Runtime documentation overclaims what receipts, payments, signatures, or public keys prove.
- Users mistake public proof for trustworthiness.
- Agent capabilities become vague authority.
- Reputation telemetry becomes a trust score.
- Memory accumulates without dispute, correction, repair, or forgiveness.
- Covenant declarations are interpreted as funded capital commitments before funding is verifiable.

## Falsification focus

This volume fails wherever the system proves that an action occurred but fails to
preserve the meaning of the commitment behind that action.

It also fails wherever HODLXXI mechanisms make users more confident without making
behavior more trustworthy, legible, repairable, or accountable.
