# Missing Runtime Mechanisms

## Forgiveness and repair

Status: not implemented.

This matters because durable cooperation cannot be only memory of failure. CRT needs
future mechanism support for repair, restitution, forgiveness, and bounded re-entry
after failure. Without repair, memory can become permanent punishment rather than
accountable continuity.

## Disputes and corrections

Status: not implemented.

Receipts can be valid while a counterparty disputes meaning, quality, acceptance, or
obligation fulfillment. A signed event may be true as event proof while remaining
contested as obligation proof.

## Delegated legitimacy

Status: not fully implemented.

Signed capabilities show what an agent claims it can do, not necessarily what it has
legitimate authority to do. Delegated authority needs explicit delegator, scope,
duration, revocation, acceptance, and verification semantics.

## First-class obligations

Status: not implemented.

Current paid jobs model request, payment, and result, but not obligation, breach,
remedy, acceptance, or restitution. Paid action is not automatically commitment unless
those terms are explicitly modeled.

## External memory anchoring

Status: not implemented or not established by Canon.

Local event-chain health is not the same as externally anchored public memory. A local
chain can support runtime audit, but Canon should not call it durable public memory
without an anchoring mechanism and verification rule.

## Counterparty model

Status: partial.

Requester public key / Human Proof exists in some flows, but CRT needs clearer
counterparty identity and proof-level semantics. Current runtime evidence can show key
control or a requester proof path; it does not yet provide a complete model of role,
authority, obligation acceptance, or legitimacy.
