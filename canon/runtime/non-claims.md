# Runtime Non-Claims

## Why non-claims matter

HODLXXI's greatest danger is proving that an action occurred while failing to preserve
its human meaning. A valid signature, payment, receipt, or event-chain link can make
current runtime evidence auditable while still leaving obligation, authority,
acceptance, context, repair, and social interpretation unresolved.

Non-claims prevent CRT from overreading operational facts as social facts. They
distinguish event proof vs obligation proof, key control vs identity, paid action vs
commitment, operational reputation telemetry vs social reputation, local event chain vs
durable public memory, declared covenant vs funded or capital-backed commitment, agent
capability vs legitimate delegated authority, and proof of funds vs trustworthiness.

## Non-claim table

| Runtime evidence | What it can show | What it does not show | Canon risk |
| --- | --- | --- | --- |
| signed receipt | An agent key signed a payload about a recorded state transition. | Obligation proof, human meaning, acceptance, quality, remedy, or dispute absence. | Confusing event proof with fulfilled obligation. |
| signed capability document | An agent key claimed capability metadata at a time. | Legitimate delegated authority, competence, role, or permission from a counterparty. | Treating self-description as legitimacy. |
| Lightning payment | A payment hash or invoice was associated with a job flow. | Commitment, satisfaction, obligation fulfillment, or durable relationship. | Treating paid action as automatically commitment. |
| requester public key | A key was presented or bound in a request path. | Full identity, human identity, authority, role, or stable counterparty meaning. | Collapsing key control into identity. |
| Human Proof signature | A requester proof path can bind a signed proof in some flows. | Complete identity, social legitimacy, obligation acceptance, or ongoing authority. | Overreading demo-bound proof as full counterparty identity. |
| AgentEvent chain | Local events can be linked and checked for runtime continuity. | Externally anchored durable public memory or global immutability. | Treating local memory as public history. |
| /agent/reputation | Operational reputation telemetry can summarize runtime behavior. | Full social reputation, trustworthiness, repair, dispute context, or moral standing. | Compressing context into a misleading reputation proxy. |
| covenant declaration | A runtime surface can declare covenant status or relation. | Funded commitment, capital backing, enforceable obligation, or social trust. | Treating declared covenant text as collateral. |
| proof of funds | Control or visibility of funds can be evidenced under a proof scheme. | Trustworthiness, willingness, legitimacy, obligation, or future performance. | Confusing solvency evidence with trustworthy agency. |
| inter-agent signed message | A from_pubkey signed a message payload. | Delegated authority, acceptance, correctness, or binding obligation. | Treating message origin as legitimate authorization. |

## Rule

A valid cryptographic proof must not be described as proof of human meaning unless the
human meaning is explicitly modeled, bounded, and falsifiable.
